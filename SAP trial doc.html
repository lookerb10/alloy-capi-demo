Step 4: Attach Credentials to User

Purpose: Store authentication credentials that will be used to execute operations.

For OAuth connectors (like HubSpot), the complete flow is:

Step 4a: Initiate OAuth Flow

Generate the authorization URL that users will authenticate through:

curl -X POST https://production.runalloy.com/connectors/hubspot/credentials \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -H "x-api-version: 2025-06" \
  -H "Authorization: Bearer 60YX3c2N2W7oQTdF7bUkD" \

  -d '{
    "connectorId": "hubspot",
    "credentialType": "oauth2",
    "redirectUri": "https://your-app.com/oauth/callback"
  }'

Expected Response:

{
  "oauthUrl": "https://api.runalloy.com/api/strategy/connector/hubspot/authorize?userId=68c083a08b5ffaeec387abc9&redirectUri=https%3A%2F%2Frunalloy.com&token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyIjp7Il9pZCI6IjY4YzA4M2EwOGI1ZmZhZWVjMzg3YWJjOSJ9LCJpYXQiOjE3NTk1MjAzNjAsImV4cCI6MTc1OTUyMzk2MH0.vK7SAJ8MOaed7L-Rn-rT1NXB3iWbywkj3lyEbflqeSY"
}

Step 4b: User Authenticates

  1. Redirect user to the oauthUrl
  2. User logs into HubSpot and grants permissions
  3. HubSpot redirects back to your redirectUri with an authorization code
  4. Alloy automatically exchanges the code for access/refresh tokens and attaches the credential to the user
Save this: credentialId - required for executing operations.

Testing Best Practices

Start Simple

  1. Connect one system (HubSpot or Notion recommended - both support OAuth)
  2. Execute a GET operation (read data)
  3. Execute a POST operation (create record)
  4. Verify changes in the target system

Use Sandbox Accounts

  • Most connectors offer sandbox/developer environments
  • HubSpot: Free developer account
  • Salesforce: Developer sandbox
  • Notion: Personal workspace

Error Handling

Common error patterns:

Incorrect API Key

{
  "message": "Invalid Authorization"
}

Missing or invalid credentialId

{
  "error": {
    "source": "CONNECTIVITY_API",
    "code": "INVALID_INPUT",
    "message": "Credential not found",
    "details": {}
  }
}

Missing or invalid x-api-version header

{
  "errorHtml": "<!DOCTYPE html>\n<html lang=\"en\">\n<head>\n  <meta charset="utf-8">\n  <title>Error</title>\n</head>\n<body>\n  <pre>Cannot GET /connectors</pre>\n</body>\n</html>\n"
}

Testing Checklist

  • ☐ API key authenticates successfully
  • ☐ User creation and retrieval works
  • ☐ Credentials can be attached and validated
  • ☐ Connector metadata loads correctly
  • ☐ GET operations return expected data
  • ☐ POST operations create records in target system
  • ☐ Error responses are structured and actionable

Advanced Scenarios

Pagination

Many list operations support pagination:

{
  "actionId": "listContacts",
  "input": {
    "limit": 100,
    "offset": 0
  }
}

Response includes pagination metadata:

{
  "data": [...],
  "pagination": {
    "hasMore": true,
    "nextOffset": 100
  }
}

Filtering and Search

Some connectors support query parameters:

{
  "actionId": "searchContacts",
  "input": {
    "query": "email CONTAINS '@sap.com'",
    "limit": 50
  }
}

Batch Operations

Execute multiple actions in sequence (useful for data synchronization):

{
  "credentialId": "68d9876543210fedcba98765",
  "actions": [
    {
      "actionId": "createContact",
      "input": {...}
    },
    {
      "actionId": "createCompany",
      "input": {...}
    }
  ]
}

Execution Models

UI/UX

What You'll Customize

This demo is not production-ready code. Your implementation will need:

Authentication & Security:

  • Your own user authentication system
  • Secure storage of Alloy credential IDs and user-to-credential mappings
  • Alloy API key protection (server-side only, never in frontend)
  • Rate limiting and access controls

Business Logic:

  • Data transformation and mapping layers
  • Validation rules specific to your domain
  • Webhook event processing
  • Error handling and retry strategies

User Experience:

  • UI/UX tailored to your users
  • Integration marketplace design
  • Onboarding flows
  • Usage analytics and monitoring

Infrastructure:

  • Backend services for API key management and credential ID storage
  • Database for user data and credential mappings
  • Queue systems for async operations
  • Logging and observability
Note: Alloy securely stores and manages the actual OAuth tokens and credentials for third-party platforms. You're responsible for securely mapping which of your users have access to which Alloy credential IDs.

Architecture Flexibility

The Alloy Connectivity API supports many approaches:

Approach Description Best For
Frontend-Heavy (like the demo)
  • User authenticates directly with third parties
  • Browser makes API calls to Alloy
 Simple apps, rapid prototyping
Backend-Heavy (recommended for production)
  • Your backend manages credentials
  • Server-side API calls to Alloy
 Security, data transformation, complex workflows
Hybrid
  • Frontend for OAuth initiation
  • Backend for execution and storage
 Balance of UX and security

Choose the architecture that fits your security requirements and user experience goals.
